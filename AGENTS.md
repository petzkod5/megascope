# AGENTS.md

Guidance for AI coding agents working in **megascope** — a self-hosted Kubernetes homepage
that auto-discovers Gateway API HTTPRoutes. One container: a Go backend (stdlib only) serves
a JSON API plus the built React/TypeScript SPA.

## Golden rules

- **Conventional Commits, always.** Every commit message MUST follow
  [Conventional Commits](https://www.conventionalcommits.org/): `type(scope): summary`.
  Releases are automated from these messages (see "Commits, PRs & releases"), so the type is
  load-bearing.
- **NEVER `git push` without explicit human permission.** Commit locally when asked; do not
  push, open PRs, or create tags/releases unless the user explicitly tells you to.
- **NEVER hand-edit versions or generated files.** `charts/megascope/Chart.yaml`
  (`version`/`appVersion`), the `version` in `frontend/package.json` + `package-lock.json`,
  and `CHANGELOG.md` are owned by release-please. `charts/megascope/README.md` is generated by
  helm-docs from `README.md.gotmpl` — edit the template, never the output.
- Keep changes **cluster-agnostic**: no hardcoded cluster, gateway, or namespace names in code
  or chart defaults (the `ghcr.io/petzkod5/megascope` image default is the one allowed owner ref).

## Layout

- `backend/` — Go API + Kubernetes discovery/health. Single `main.go`, stdlib only.
- `frontend/` — React 19 + TypeScript + Vite SPA (the command-center UI).
- `charts/megascope/` — parameterized Helm chart (+ helm-docs README, values.schema.json).
- `.github/workflows/` — `ci.yml` (lint/test), `build.yml` (image + chart publish), `release.yml` (release-please).
- `Dockerfile` — multi-stage: build frontend → build backend → distroless non-root image.

## Build, test & run

Run the same checks CI runs before declaring work done.

Backend (`cd backend`):
- Vet: `go vet ./...`
- Test: `go test ./...`
- Build: `go build ./...`
- Format: `gofmt -w .` (Go standard; CI does not enforce it, but keep diffs gofmt-clean)

Frontend (`cd frontend`):
- Install: `npm ci` (use `npm install` only when intentionally changing dependencies)
- Build + type-check: `npm run build` (runs `tsc --noEmit` then `vite build`) — this is the
  only type/lint gate; there is no separate ESLint or Prettier.
- Dev server: `npm run dev` (proxies `/api` → `http://localhost:8080`)

Chart (repo root):
- Lint: `helm lint charts/megascope`
- Render: `helm template megascope charts/megascope --namespace megascope`
- After editing `values.yaml` or `README.md.gotmpl`, regenerate the chart README:
  `helm-docs --chart-search-root=charts --template-files=README.md.gotmpl`

Run the whole app locally (needs a reachable cluster):

```bash
kubectl proxy --port=8001                                   # terminal A
cd backend && KUBE_API=http://localhost:8001 go run .       # terminal B (:8080)
cd frontend && npm install && npm run dev                   # terminal C
```

Without `KUBE_API` the backend still serves but discovery is disabled (empty portal) — fine
for UI-only work.

## Backend conventions

- **Stdlib only — no third-party Go dependencies.** `go.mod` lists none; it talks to the
  Kubernetes API directly with the mounted ServiceAccount token (no client-go). Do not add deps.
- **Read-only cluster access:** the backend only ever GETs from the cluster (discovery +
  health summary) and probes routes with plain outbound HTTP `GET`s. Never add write/mutate calls.
- Config is env vars read in `main()` via `envOr`: `ADDR` (`:8080`), `WEB_DIR` (`./web`),
  `CLUSTER_NAME` (`production`), `PROBE_HEALTH` (`true`), `POLL_SECONDS` (`30`), `KUBE_API` (unset).
- HTTP endpoints: `GET /api/routes` (state JSON), `GET /healthz`, `GET /metrics` (Prometheus
  text exposition, hand-rolled — no client_golang).
- Tile annotations use the `app.megascope.io/` prefix (`name`, `group`, `icon`, `url`, `hidden`).
- Tests are table-driven with `net/http/httptest`; add cases to `main_test.go` next to the
  code they cover.

## Frontend conventions

- TypeScript is **strict** (`noUnusedLocals`, `noUnusedParameters`, `noFallthroughCasesInSwitch`);
  keep `npm run build` clean.
- Styling is **inline `style={{}}` using the CSS custom properties** defined in `src/index.css`
  (e.g. `--bg-*`, `--border-*`, `--text-*`, `--accent`, `--font-mono`, `--radius-*`, `--shadow-*`).
  No CSS framework and no CSS-in-JS library — match the existing components.
- Icons go through the `Ico` component (`lucide-react`). Common icons are statically imported in
  `Ico.tsx`; others lazy-load. Add frequently-used icons to its `STATIC` map.
- Reusable primitives live in `src/components/`; screens in `src/views/`. Reuse existing
  primitives (`Panel`, `Button`, `Badge`, …) before adding new ones.
- Avoid adding frontend dependencies; the bundle is kept deliberately small.

## Commits, PRs & releases

- **Conventional Commits** drive versioning. Type → release effect:
  - `feat:` → minor (0.1.0 → 0.2.0); `fix:` / `perf:` → patch (0.1.0 → 0.1.1).
  - `feat!:` or a `BREAKING CHANGE:` footer → minor while pre-1.0 (the project stays in 0.x).
  - `chore:`, `ci:`, `docs:`, `refactor:`, `test:`, `build:`, `style:` → no release.
  - Use a scope for the area: `backend`, `frontend`, `chart`, `ci`, `deps`.
- PR titles must also be conventional — squash-merge uses the PR title, and a CI job validates it.
- Releases are automated by `release.yml` (release-please): conventional commits merged to `main`
  open a "release PR" that bumps versions + `CHANGELOG.md`; merging that PR tags `vX.Y.Z`, creates
  the GitHub Release, and publishes the image + chart. Never tag or cut a release by hand.
- `CONTRIBUTING.md` is the human-facing version of these conventions.

## Safety

- Do not push, force-push, tag, or create releases/PRs without explicit permission (see Golden rules).
- Do not commit secrets; `.env*` is gitignored. No secrets are needed locally beyond `kubectl proxy`.
- Treat unexpected working-tree changes as the user's in-progress work; ask before reverting them.
